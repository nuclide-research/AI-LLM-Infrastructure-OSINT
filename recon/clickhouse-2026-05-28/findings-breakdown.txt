WHAT IS EXPOSED

Apptica's production ClickHouse database cluster node is open to the public
internet with no authentication. The ClickHouse /play browser SQL interface
is accessible to anyone with a web browser. No credentials required.

Apptica is a commercial app store intelligence platform. They sell revenue
estimates, download data, and advertising intelligence for mobile apps.
Their product is built on the data stored in this database.

The exposed data includes over 700 billion rows across the primary tables:

  revenue_hist_ml_v2:         138 billion rows (their ML model training data)
  revenue_hist_ml_v2_final:    93 billion rows (finalized predictions)
  revenue_hist_ml_v1:          69 billion rows (previous model version)
  offers_log_daily:            66 billion rows (advertising spend data)
  application_ratings:         30 billion rows (app store ratings history)
  top_history:                  9 billion rows (chart position history)
  usage_metrics_dau:            1 billion rows (daily active user metrics)

WHY IT MATTERS

The revenue_hist_ml tables are Apptica's competitive moat — the ML model
that generates their revenue estimates is trained on this data. A competitor
who queries this database gets the training data, the model inputs, and the
predicted outputs. The intelligence product that Apptica charges for is built
from data that is currently readable by anyone.

The offers_log_daily table contains advertising spend records with
advertiser and publisher app IDs, ad network IDs, and a key_id field
(Nullable String) that appears to be an advertiser API key or campaign
identifier. This is advertising intelligence at scale: 66 billion records
showing which apps advertised in which networks, how much, and where.

The cluster was actively receiving INSERT operations during enumeration.
This is a live production system, not a test environment.

HOW WE KNOW IT IS APPTICA

The database name is apptica_stat. The table naming conventions (AP_4085,
AP_4600, AP_3761, AP4579 — internal ticket numbers) match a production
codebase. The cluster node name ch-s01-r02 (server 1, replica 2) shows
this is part of a replicated production cluster, not a standalone dev instance.
The query log shows INSERT operations with columns including developer_name
— the kind of production write traffic that only happens on a live system.

WHAT AN ATTACKER GETS

  SELECT app_id, geo_id, downloads, revenue FROM apptica_stat.revenue_hist_ml_v2
  WHERE app_id = <competitor_app_id> -- dump revenue estimates for any app

  SELECT DISTINCT advertiser_application_id, adnetwork_id, key_id
  FROM apptica_stat.offers_log_daily
  WHERE key_id IS NOT NULL LIMIT 1000 -- surface advertiser API keys

  SELECT * FROM apptica_stat.creative_scores
  ORDER BY score DESC LIMIT 100 -- top performing ad creatives

REMEDIATION

Set CLICKHOUSE_PASSWORD environment variable or add <password> to the
default user in config.xml. Restrict listen_host to internal IPs only.

POPULATION-LEVEL FINDING
Scope: 200 hosts, 5,243 total population
===========================================================================

ClickHouse has an 8% unauth rate in the public-internet-facing population.
This is better than Ollama (~0% auth configured) and worse than Argo Workflows
(100% auth). The improvement over Ollama reflects ClickHouse's Q1 2025 change
to their default Docker image, which disabled network access for the default
user following exposure incidents tied to the DeepSeek deployment wave.

The 8% unauth tail consists of older self-managed deployments that predate
the Docker image change or that override defaults via custom config.

Every single one of the 200 probed hosts leaks their internal hostname via
the X-ClickHouse-Server-Display-Name HTTP response header, regardless of
authentication state. This is systematic information disclosure at scale.
Two hosts reveal Kubernetes ClickHouse Operator cluster topologies via their
pod DNS names in this header.
