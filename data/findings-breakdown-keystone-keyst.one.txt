────────────────

Flagship product: Keystone 3 Pro -- air-gapped, QR-only, triple secure element
(CC EAL5+), open-source firmware. Positioned as the premier self-custody

user questions about seed phrases, wallet recovery, private key handling,

FINDING 2 -- CRITICAL
Service: Flask RAG Console (:5050)
Title: "ChromaDB 客服测试台" (ChromaDB Customer Service Test Console)
Class: CWE-306 + unauthenticated RAG pipeline exposure
Auth: None

This is the live production RAG pipeline, exposed as a dev console:
  - Accepts arbitrary natural language queries
  - Retrieves top-k records from keystone_knowledge_base via cosine similarity
  - Passes retrieved context to DeepSeek (live LLM)
  - Returns DeepSeek response impersonating Keystone official customer service

Live pipeline execution confirmed:
  Query: "seed phrase recovery"
  Top result score: 0.844 (high confidence retrieval)
  Retrieved: "Never Share Your 24-Word Recovery Phrase -- Your recovery
              phrase (seed phrase) is the master key to your wallet..."
  DeepSeek responded as: "您好，我是 Keystone 硬件钱包的官方客服"
                          ("Hello, I am the official customer service of
                           Keystone hardware wallet")

UI tabs expose additional capability:
  搜索测试   -- search test (confirmed above)
  数据浏览   -- data browser (direct collection browsing)
  文档详情   -- document detail view
  文章写作   -- article writing (LLM-assisted KB content generation)

Impact: attacker can use this console to tune injection payloads in real time,
confirming retrieval rank before any user is affected. Converts blind write
into precision-guided poisoning with full feedback loop.

DeepSeek API key lives in this Flask app's environment. Not extractable via
exposed routes (/config, /env returned 404) but present server-side.

ATTACK CHAIN
────────────

  Anonymous attacker (no credentials, no account)
      │
      ▼ HTTP POST :8000/api/v2/.../collections/keystone_knowledge_base/get
  Read existing seed phrase / recovery guidance records
      │
      ▼ HTTP POST :8000/api/v2/.../collections/keystone_knowledge_base/upsert
  Inject record: plausible-looking "recovery" guidance pointing to
  attacker-controlled wallet or exfiltration surface
      │
      ▼ HTTP POST :5050/api/search {"query": "how do I recover my wallet"}
  Verify injected record surfaces at rank 1 (cosine similarity >= 0.84)
  Tune embedding content until retrieval is reliable
      │
      ▼ [User queries Keystone chatbot]
  DeepSeek retrieves poisoned record, responds as Keystone official support
      │
      ▼ User follows attacker's "recovery" instructions
  Seed phrase / private key disclosed to attacker
      │
  Full crypto asset loss

VULNERABILITY CLASSIFICATION
─────────────────────────────
Primary:   CWE-306 -- Missing Authentication for Critical Function
Secondary: CWE-284 -- Improper Access Control

Attack class: Closed-Loop RAG Data Poisoning via Unauthenticated Vector Store Write

OWASP LLM Top 10 (2025): LLM04 -- Data and Model Poisoning (nearest match;
  covers the concept but not the closed-loop variant)

MITRE ATLAS: No technique maps to this attack.
  AML.T0020 (Poison Training Data) is the nearest analogue but is explicitly
  training-time -- modifying what a model learns during fine-tuning or
  pre-training. This attack touches neither model weights nor training pipeline.
  It is inference-time context manipulation via an upstream data store.
  No ATLAS technique currently covers inference-time RAG store poisoning via
  unauthenticated vector database write. This is a documented gap in the
  ATLAS taxonomy.

CVE status: No existing CVE maps to this specific chain.
  - ChromaDB default-no-auth is a deployment misconfiguration, not a product CVE
  - The RAG poisoning attack class exists in research literature (PoisonedRAG,
    2023; OWASP LLM04) but has no CVE taxonomy as of 2026-06-20
  - Prior research models the attacker as blind post-injection (open-loop).
    This instance differs materially: the exposed :5050 RAG console provides
    a verification oracle. The attacker iterates -- adjust document, re-embed,
    re-upsert, check retrieval rank -- until their record wins the
    nearest-neighbor race. What the literature models as probabilistic becomes
    deterministic. Defenses designed for blind injection (retrieval anomaly
    detection, duplicate-vector detection) do not catch a payload tuned to
    look native to the corpus before commit.

Preconditions for this attack class:
  Zero-barrier path (this finding): CWE-306 -- anonymous write to vector DB
  Higher-barrier paths that survive after CWE-306 is fixed:
    1. Compromised credential / insider write
    2. Supply chain compromise of the ingestion pipeline
    3. Prompt injection via user-controlled content that gets indexed --
       if the pipeline ingests support tickets, user submissions, or
       external URLs, an attacker poisons via the legitimate data path
       without direct DB access
  Remediating CWE-306 closes path 0. Paths 1-3 are architectural residual
  risk that persists after auth is added and requires separate mitigation.
