THESIS CONTRIBUTION
------------------------------------------------------------
Meilisearch is a single-env-var auth platform: MEILI_MASTER_KEY off by
default but enabled by ~90% of LIVE operators in production. This is
the OPPOSITE shape of ComfyUI (77.5% unauth) and consistent with the
auth-on-default thesis corollary:
    > Defaults are load-bearing, but documented single-step opt-ins
    > approach the same end state at population scale.
Meilisearch ships docs that prominently feature the master_key step;
operators do it. ComfyUI has no master_key concept; operators do not
add one (would require external reverse proxy). The framework's choice
of WHETHER auth exists at all is the load-bearing decision; once an
opt-in exists, documented as the production path, operators converge.

THIS IS A POPULATION-SCALE EVIDENCE POINT FOR INSIGHT #13 IN ITS
TRIPLE-PLATFORM FORM:
  - Langfuse: forced auth -> 0% unauth (the ideal)
  - Meilisearch: opt-in documented as prod path -> 10% unauth
  - Phoenix: opt-in not foregrounded -> 25% unauth
  - ComfyUI: no auth concept -> 78% unauth (the worst case)

INSIGHT CANDIDATE #88
------------------------------------------------------------
"Auth gradient by config friction." Across four platforms with comparable
operator pools, the platform's auth friction at deploy time predicts the
population-scale unauth rate more strongly than any specific config flag:
  0 steps  (forced) -> ~0%   (Langfuse)
  1 env var (doc-foregrounded) -> ~10%  (Meilisearch)
  1 env var (config-flag, not foregrounded) -> ~25%  (Phoenix)
  no auth concept -> ~78%  (ComfyUI)
The function is roughly linear in friction-steps but the slope changes at
step-0 (forced) and step-infinity (no concept). Promote on a fifth platform
that lands in the same gradient.
