  GET /api/agents                       -> 200, 0 agents
                                        (default Khoj install has no
                                        custom personas)

  /api/user                             -> 404
      (this Khoj version routes user under /api/health rather than the
      canonical /api/v1/user)

Reverse DNS suggests obsidian-integrated personal deployment (the-judsons.com).
This is the Khoj "easy-mode" failure: user installed for personal RAG, did
not set up auth, exposed instance to the public internet on port 443. Any
visitor can read the user's chat history and indexed documents via the
documented Khoj API without credentials.

We did not exercise these reads. Verification stopped at proving the
unauth surface is open (HTTP 200 on /api/chat/sessions + non-empty content
types). The next step on a permissive instance would be
/api/chat/history?conversation_id=<id> -> message bodies -> raw indexed
documents — every one of these is auth-bypassed once /api/health returns
default@example.com.


VERSION DISTRIBUTION
--------------------
None of the 7 confirmed instances exposed a parseable version string on
/api/health. Khoj's /api/health endpoint returns either:
  {"detail":"OK"}                       (auth-required mode)
  {"email":"<user>"}                    (permissive / anon mode)

Version is in the user object at /api/v1/user (which exists on newer
Khoj builds: keys include khoj_version, has_documents, photo, etc) —
we did not exfiltrate version strings from authenticated endpoints, in
line with the restraint posture.

THE THESIS
----------
Khoj is single-user-by-default. The "anonymous mode" is the failure
mode in a multi-user world: a Khoj instance with auth not configured
hands the entire instance to whichever HTTP client shows up first.
The Khoj docs say so explicitly — production deployments need
KHOJ_DEBUG=False, anonymous_mode disabled, and either Google OAuth
or Magic Link auth configured. Operators who skip this expose:

  - All chat conversations (search + retrieval history)
  - All indexed personal documents (Notion / Obsidian / GitHub /
    PDF / etc.)
  - All custom agent personas (which leak the user's interests)
  - All scheduled automations (which leak the user's routines)

This is structurally identical to Open-WebUI's `WEBUI_AUTH=False`
failure and LibreChat's `ALLOW_REGISTRATION=true` failure — three
generations of AI-assistant UIs all share the "default-permissive
for solo deployment" tradeoff, and all three leak personal data
when the operator forgets to flip the flag for production.

Of 5 unique Khoj IPs observed on the internet today, 1 (20%) was
in the anonymous failure mode. Small-N sample, but consistent with
the broader auth-on-default investigation: hardened defaults are
necessary but not sufficient; population-level visibility shows
real users continue to ship the easy-mode configuration.
