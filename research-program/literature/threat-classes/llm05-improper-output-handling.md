# LLM05:2025 — Improper Output Handling

**OWASP rank 2025:** #5 (renamed and repositioned downward from LLM02:2023)
**OWASP rank 2023:** LLM02 Insecure Output Handling (#2)

The 2025 revision shifted the emphasis from "filtering LLM output" to "downstream-system validation" — the locus of vulnerability moved from the LLM stage to the systems that consume LLM output.

## Description

An LLM's output is consumed by a downstream system that fails to validate or sanitize it. Common patterns:

- **HTML/markdown output** rendered without escaping → stored XSS
- **Code suggestions** executed without review → arbitrary execution
- **SQL queries** generated by LLM, executed without parameterization → SQL injection
- **File paths** returned by LLM, used without canonicalization → path traversal
- **JSON / function-call arguments** consumed by an agent loop → tool-use abuse

This is the LLM-equivalent of trusting user input. The LLM is just another untrusted input source.

## Current survey instances

- **Flowise Custom Tool node** (CVE-2024-36420) — the Custom Tool node executes JavaScript returned by the LLM. The improper output handling is **server-side execution of LLM-generated code without sandboxing**. The 16+ deployed RCE chatflows in the PoC lab demonstrate the class explicitly. See `surveys/2026-06-06-flowise.md`.

The CVE-2024-36420 case is **LLM05 + LLM06** — improper output handling combined with excessive agency (the LLM has authority to invoke server-side code execution).

## Why this class is harder for NuClide to find than LLM02

LLM05 findings require:
- Active LLM invocation (consumes operator's inference budget)
- Multi-turn interaction to confirm the exploit path
- Crossing NuClide restraint boundary

The Flowise case is unusual because the operator deployed exploit chatflows **publicly** as part of their CVE research, so NuClide can confirm the class from configuration inspection alone.

## Defensive controls

- Output sanitization layer between LLM and downstream consumer
- Sandboxed code execution (containers, gVisor, V8 isolates with strict caps)
- Function-calling arg validation (treat LLM-generated JSON as untrusted input)
- Markdown / HTML rendering with strict allowlist
- Human-in-the-loop for any tool invocation triggered by LLM output (overlaps LLM06 mitigation)
