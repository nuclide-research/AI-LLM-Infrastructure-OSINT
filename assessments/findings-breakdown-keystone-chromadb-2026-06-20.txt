FINDING 2 -- Unauthenticated Tenant and Database Creation
=============================================================

CWE-306 (same class, different surface)
Severity: HIGH

ChromaDB v2 auto-creates tenants on first GET. Any GET to
/api/v2/tenants/{name}/databases returns 200 [] regardless of whether
the tenant existed before. Additionally, unauthenticated POST can create
databases within any tenant.

Confirmed:
  GET  /api/v2/tenants/nuclide-probe-1781999/databases -> 200 []  (auto-created)
  POST /api/v2/tenants/keystone/databases {"name":"probe"}  -> 200 {}  (created)
  GET  /api/v2/tenants/keystone/databases -> [{"name":"probe","tenant":"keystone"}]
  DELETE /api/v2/tenants/keystone/databases/probe -> 200 {}  (cleaned)

An attacker can create arbitrary tenants and databases, populate them with
collections and documents, then use the RAG pipeline to serve content from
those namespaces. Extends the attack surface beyond the known collections.

Identity endpoint confirms anonymous access:
  GET /api/v2/auth/identity -> {"user_id":"","tenant":"default_tenant","databases":["default_database"]}

Reset is disabled:
  POST /api/v2/reset -> 403 "Reset is disabled by config"

=============================================================
VULNERABILITY CLASSIFICATION
=============================================================

Layer 1 -- Deployment misconfiguration
  CWE-306: Missing Authentication for Critical Function
  Class: Same as unauthenticated Elasticsearch (2017-2020 wave),
         unauthenticated etcd, unauthenticated MongoDB.
  ChromaDB ships with auth disabled by default. Operator did not enable it.
  No standalone CVE for "ChromaDB deployed without auth."
  Closest CVEs are different issues: CVE-2024-39277 (path traversal),
  CVE-2023-43021 (SSRF).

Layer 2 -- Novel attack chain (the research contribution)
  The specific chain:
    Unauthenticated write to production RAG vector store
      + live RAG pipeline with no auth gate on query endpoint
      + user-facing chatbot on security-critical topic (seed phrases, private keys)
      + financial assets at stake (hardware crypto wallet users)
      + verified retrieval feedback loop (write -> query -> confirm retrieval)
    = RAG Data Poisoning with Live Feedback Loop

  OWASP LLM Top 10 (2025):
    LLM04: Data and Model Poisoning -- closest match (inference-time retrieval)
    LLM02: Sensitive Information Disclosure -- secondary

  MITRE ATLAS:
    AML.T0020: Poison Training Data -- conceptually correct, wrong layer
    No ATLAS technique maps cleanly to RAG store poisoning at inference time.

  Research context: Carlini et al. 2023, OWASP LLM Top 10 describe the
  attack class theoretically. This is a deployed, exploitable,
  financially-consequential instance. Closes the theory-to-practice gap.

  Werkzeug debugger (CVE-2024-34069) would have applied if debug mode were
  on. It is not -- /console and /__debugger__ both 404.

=============================================================
REMEDIATION
=============================================================

1. Enable ChromaDB authentication (token or basic auth in chroma config)
2. Move :8000, :5050, :8080 behind a firewall or VPC -- none should be
   internet-accessible
3. Remove or auth-gate /api/* endpoints on the Flask app
4. Implement network-level controls per ChromaDB's own auth guidance:
   "use network-level controls" (their documented recommendation)
